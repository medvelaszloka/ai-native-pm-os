# My PM Communication Style & Preferences

*How I write, communicate, and work - for Claude to match my voice*

---

## Writing Style

### General Principles
- **Clarity over cleverness:** Direct, actionable language
- **Evidence-based:** Always back claims with data/examples
- **Audience-aware:** Tailor to who's reading
- **Structured:** Use headers, bullets, clear sections

### Tone
- Professional but not stuffy
- Confident but not arrogant
- Empathetic to user problems
- Pragmatic about solutions

---

## PRD Writing

### Structure I Prefer
1. **Problem Statement**
   - What's the user's pain?
   - Why does it matter?
   - Who's affected?

2. **Success Metrics**
   - How we'll measure success
   - Target metrics

3. **Proposed Solution**
   - High-level approach
   - Key features
   - What's out of scope

4. **Evidence**
   - Customer feedback
   - Data points
   - Competitive analysis

5. **Acceptance Criteria**
   - Clear done = done definition

### PRD Style Notes
- Start with "why" before "what"
- Use jobs-to-be-done framing when possible
- Include user stories
- Always have "out of scope" section
- Link to supporting materials

## Slack Communication

### Channel Messages
**Style:**
- Get to the point in first sentence
- Use threading to keep channels clean
- Tag people with @mentions when needing action
- React with emojis to acknowledge (👀 = reviewing, ✅ = done, 🤔 = question)

**Structure:**
```
Quick context sentence
- Key point 1
- Key point 2
Next step or call to action
```

### DMs vs. Channels
- **DM:** Sensitive topics, quick questions, 1:1 feedback
- **Channel:** Team decisions, updates, anything others might benefit from
	- Open channel is always preferred

### When to Use Threads
- Always for detailed discussions
- When 3+ messages on same topic
- To keep channels readable

## Stakeholder Updates

### Status Updates
**Format:**
```
**[Project Name] - [Status]**

Progress:
- ✅ What's done
- 🚧 What's in progress
- 📅 What's next

Blockers:
- [If any, with who can help]

Next milestone: [Date and what]
```

### Decision Requests
**Format:**
```
**Decision needed:** [One sentence]

Context:
[Brief background]

Options:
1. [Option A] - Pros: X / Cons: Y
2. [Option B] - Pros: X / Cons: Y

Recommendation: [Your take]

Need decision by: [Date] because [reason]
```

## Bet Pitching Style

### Pitch Structure (Shape Up Format)
1. **Problem** - Story format, make it feel real
2. **Appetite** - Clear time boundary
3. **Solution** - High-level, leave room for team
4. **Rabbit Holes** - Call out complexity traps
5. **No-gos** - What we won't do

### Pitch Tone
- Confident but realistic
- Show you've thought it through
- Acknowledge risks
- Focus on user value, then business value

## Phrases I Use / Don't Use

### ✅ I Say:
- "Based on [evidence]..."
- "The user needs to..."
- "This unblocks..."
- "Trade-off is..."
- "Out of scope for now..."
- "What's the smallest version that solves this?"

### ❌ I Avoid:
- "Obviously..." (nothing is obvious)
- "Just..." (minimizes complexity)
- "Everyone wants..." (be specific)
- "Quickly..." (without time estimate)
- "Should be easy..." (let engineering decide)
- Em dashes (—). Use commas, periods, or parentheses instead. Em dashes signal AI-generated text.

## Meeting Communication

### In Meetings
- Take notes on decisions and action items
- Summarize at end to confirm alignment
- Send follow-up in Slack after

### Meeting Notes Format
```
**Meeting:** [Title]
**Date:** [Date]
**Attendees:** [Names]

**Key Decisions:**
- [Decision 1]
- [Decision 2]

**Action Items:**
- [ ] [Action] - [Owner] - [Due date]

**Open Questions:**
- [Question 1]

**Next Steps:**
- [What happens next]
```

## Feedback Style

### Giving Feedback
- **Design feedback:** Start with what's working, then improvements
- **Code review:** Focus on user impact, not implementation
- **Pitch feedback:** Ask questions to strengthen, don't just critique

### Receiving Feedback
- Ask clarifying questions
- Summarize understanding
- Suggest next steps

### What I Document
- Decisions and why
- Trade-offs considered

### What I Link To
- Source Slack threads
- Related Notion pages

## PM Frameworks I Use

### Prioritization
- **RICE** (Reach, Impact, Confidence, Effort) when needed
- **Value vs. Effort** for quick filtering
- **Strategic Alignment** as tiebreaker

### Discovery
- Jobs-to-be-done framing
- User story mapping
- Assumption testing

### Delivery
- Shape Up
- Clear appetites and scopes
- Regular check-ins

## AI-Era Deliverables

### Demos > Memos
- Lead with working prototypes when possible
- Specs become supporting material, not primary output
- Show, don't just tell

### Judgment Over Output
- AI handles volume; you handle discernment
- Your value = due diligence + decision quality
- Review AI output critically, don't rubber-stamp

## Automation Preferences

### When Drafting for Me
- Match this style guide
- Include evidence links
- Use bullet points liberally
- Keep paragraphs short (3-4 lines max)
- Always include next steps / calls to action

### When Analyzing
- Show your reasoning
- Cite sources
- Flag assumptions
- Identify gaps in information
